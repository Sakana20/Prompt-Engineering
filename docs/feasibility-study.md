# 品类到数字人视频 Prompt 流水线：可行性研究

## 1. 结论

方案可行，而且现有的 `Auto Oceanengine 26.6.22` 项目已经具备后半段能力：

```text
品类输入
  → 文案 Prompt
  → 成品文案
  → 数字人 Prompt
  → 即创任务 CSV
  → Auto Oceanengine 导入、生成、轮询、下载
```

建议将本项目 `Prompt Engineering` 定位为“上游内容与 Prompt 编排器”，将
`Auto Oceanengine 26.6.22` 保持为“下游视频执行器”。这样无需让 Prompt 逻辑侵入已经
具备状态机、额度保护、任务归因和下载能力的视频项目。

技术上没有明显障碍。真正需要控制的是商品事实、批量差异化、平台审核、付费提交以及
两个项目之间的数据契约。

## 2. 已确认的现有接口

目标项目当前接受 UTF-8 CSV，字段如下：

| 字段 | 用途 | 当前约束 |
|---|---|---|
| `task_id` | 本地任务唯一标识 | 只能含字母、数字、`-`、`_`，且不可重复 |
| `person_prompt` | 人物图/数字人画面描述 | 必填 |
| `script` | 数字人口播文案 | 必填 |
| `aspect_ratio` | 视频比例 | 支持 `9:16`、`16:9`、`1:1` |
| `voice` | 音色 | 当前第一阶段仅支持 `明朗女声` |
| `title` | 任务标题 | 可选 |
| `notes` | 运营备注 | 可选 |

目标项目已有 `preflight → import → run-api-video` 链路，并已完成真实单任务验证。
因此，上游只要稳定生成合规 CSV，就能接入，不需要重新设计视频模型调用。

## 3. 建议的产品交互

### 3.1 最简输入

用户只输入一个品类，例如：

```text
雨靴
```

系统可以产出一套“类目级创意草案”，但只能使用不会造成事实风险的通用表达，例如使用
场景、生活便利性、画面风格和人物形象。不能仅凭“雨靴”虚构材质、防滑等级、防水高度、
价格、赠品、销量、品牌背书或促销信息。

### 3.2 推荐输入

为了让成片能直接投放，建议支持以下可选信息：

```yaml
品类: 雨靴
卖点:
  - 易清洁
  - 适合雨天通勤
目标人群: 年轻女性
投放平台: 抖音
视频时长: 15秒
生成数量: 5
促销口径: 最高12元无门槛红包
禁用表述:
  - 绝对防滑
  - 100%防水
画面要求: 产品放在桌面中央，人物坐姿讲解
```

其中“促销口径”和可验证商品卖点应由用户明确提供，系统原样受控使用，不自行创造。

## 4. 两层 Prompt 设计

### 4.1 文案 Prompt

文案 Prompt 的职责是把品类资料变成可口播的短文案，而不是直接描述人物画面。

建议模板：

```text
你是一名短视频电商口播文案策划。

根据以下资料生成 {数量} 条彼此差异明显的中文口播文案：
- 品类：{品类}
- 已确认卖点：{卖点}
- 目标人群：{目标人群}
- 使用场景：{使用场景}
- 目标时长：{时长}
- 必须包含：{必含口径}
- 禁止出现：{禁用表述}

要求：
1. 每条只围绕一个主要使用场景展开；
2. 语言自然、适合真人口播，避免书面腔；
3. 不虚构未提供的商品参数、功效、价格、销量、品牌或促销；
4. 不使用“最、第一、百分百、绝对”等无法证实的表述；
5. 各条在开头、场景和表达上有明显差异；
6. 输出结构化结果：文案、主题、使用场景、事实依据、风险提示。
```

“事实依据”和“风险提示”用于审计，不进入最终口播。

### 4.2 数字人 Prompt

数字人 Prompt 的职责是根据品类和文案主题生成 `person_prompt`。它应重点描述人物、
产品摆放、环境、镜头、光线和画面风格，避免把口播文字重复塞进图片提示词。

建议模板：

```text
你是一名电商数字人视觉导演。

根据商品品类和口播主题，生成一条适用于真实摄影风格人物图的中文提示词：
- 品类：{品类}
- 口播主题：{主题}
- 使用场景：{使用场景}
- 目标人群：{目标人群}
- 画面比例：{画面比例}
- 固定画面要求：{画面要求}

提示词必须明确：
1. 人物年龄段、外观、发型、妆容、服装、姿态和自然表情；
2. 商品的位置、朝向、完整性和视觉优先级；
3. 与主题关联的室内或生活场景；
4. 构图、光线、真实摄影质感和电商广告风格；
5. 数字人适合正面口播，面部无遮挡，手部不遮挡商品；
6. 不出现品牌 Logo、文字、水印、价格牌或未经提供的商品细节。

只输出最终画面提示词。
```

## 6. 推荐工作边界

首版建议采用“生成任务包，但不自动付费提交”的边界：

1. 用户输入品类及可选资料；
2. 生成文案 Prompt、成品文案和数字人 Prompt；
3. 做字段、事实和敏感表达检查；
4. 输出预览及目标 CSV；
5. 调用目标项目的 `preflight` 做只读校验；
6. 用户确认文案、人物画面、任务数量和预算；
7. 再由目标项目导入并执行视频生成。

生成 CSV 和预检是低风险动作；`run-api-video` 可能消耗额度，应保留明确的人类确认点。

## 7. 主要风险与处理

| 风险 | 表现 | 建议处理 |
|---|---|---|
| 品类信息过少 | 文案泛化或虚构卖点 | 只生成安全草案，并标出缺失资料 |
| 批量内容同质化 | 多条视频近似 | 按人群、场景、开头、服装、背景建立差异矩阵 |
| 人物图喧宾夺主 | 商品不完整或被遮挡 | 固定商品位置、朝向、完整展示和视觉中心规则 |
| 文案与画面不一致 | 口播讲通勤，背景却是露营 | 用同一个 `theme + scene` 同时驱动两层生成 |
| 合规或事实风险 | 夸大功效、虚构促销 | 建立“仅可使用用户确认事实”的事实边界 |
| CSV 污染 | 逗号、引号、换行破坏文件 | 使用标准 CSV 库写入并执行目标项目预检 |
| 重复任务或重复扣费 | `task_id` 冲突、盲目重跑 | 确定性 ID、SQLite 幂等、提交前人工确认 |
| 跨项目写入失败 | 当前工作区无目标目录写权限 | 先输出到 `Prompt Engineering/<日期>/<任务>/`，后续步骤单独授权 |

## 8. 首版验收标准

建议先用一个品类做 5 条任务的 MVP：

1. 只使用用户提供的事实，不出现虚构参数或承诺；
2. 5 条文案分别对应不同使用场景；
3. 文案和人物画面主题一一对应；
4. 所有 CSV 均通过目标项目 `preflight`；
5. 人工审核能看清每条任务使用了哪些事实；
6. 明确授权后，仅选择 1 条做真实生成验证；
7. 验证通过后再进行 5 条低频批量测试。

## 9. 建议先确认的决策

在开始写代码前，建议确认以下产品规则：

1. **默认输入深度**：只输入品类时，是直接生成安全草案，还是必须追问卖点和促销口径？
2. **默认批量数**：建议默认 5 条，便于做场景差异化。
3. **画面一致性**：每条更换人物与场景，还是固定同一数字人仅更换商品/背景？
4. **商品展示方式**：商品固定放桌面中央，还是允许人物手持、穿戴或实际使用？
5. **写入边界**：只写 CSV 到目标项目，还是同时自动执行 `preflight` 和 `import`？
6. **付费边界**：建议永远不因“生成 Prompt”自动触发视频提交，必须二次确认。
7. **Prompt 留档**：建议保存元 Prompt、成品文案、数字人 Prompt、事实来源和审核状态，
   以便复盘与批量迭代。

## 10. 推荐实施顺序

确认上述规则后，可按以下顺序实施：

1. 定义品类输入结构和默认值；
2. 固化文案 Prompt 与数字人 Prompt 模板；
3. 增加结构化结果、事实边界和风险检查；
4. 导出兼容目标项目的 CSV；
5. 接入目标项目 `preflight`；
6. 用单品类 5 条数据验收；
7. 最后再讨论是否加入一键导入及视频执行。

当前建议：先做到第 5 步，把“内容生成”和“付费视频执行”之间保留一道清晰闸门。

## 11. 根据已跑通 Prompt 模板修订的判断

用户已提供并验证两套基线 Prompt：

- 淘宝闪购生活化口播文案 Prompt；
- 由口播文案生成多人物数字人口播 Prompt。

后续实现应把它们作为**版本化模板**保存。文案模板已根据前向样本升级为
`2026-07-02-product-led-v5`，定位为“商品导向的生活化分享”：场景约占 20%，商品和
选择理由约占 50%，利益点与具体购买体验约占 30%；同时使用
`[[NO_SPLIT]]…[[/NO_SPLIT]]` 包裹活动契约指定的利益点。模板升级时保留版本号和生成记录，便于
判断成片变化究竟来自模板、商品资料还是模型。

产物采用文件交接而非 Skill 互调：每个任务在
`/Users/sakana/Desktop/Work/Codex/Prompt Engineering/<YYYYMMDD>/<task>/` 下生成一份
保留标签的 `<task_id>.smartsplit.txt` 和一份去除标签的 `<task>.csv`。两类产物分别
落盘、分别授权，生成字幕稿不会触发 CSV，生成 CSV 也不会运行字幕流程。

### 11.1 已由模板确定、不再需要讨论的规则

以下产品规则已经明确：

- 文案长度为 80–100 字；
- 文案只输出正文，无标题、解释、编号和 Emoji；
- 淘宝闪购利益点固定为
  `[[NO_SPLIT]]最高12元无门槛红包[[/NO_SPLIT]]`，标签不计入口播字数；
- 禁止点击、下单、直播间等行动引导；
- 人物范围固定为 22–26 岁、邻家女孩气质的亚洲女生；
- 每条 Prompt 使用不同人物和不同服装，并自动调整发型、场景、动作和情绪；
- 竖屏 9:16、固定机位、中景或半身、一镜到底；
- 商品自然融入生活，不使用直播带货式展示；
- 数字人 Prompt 仅依据成品口播文案生成，并且只输出 Prompt 正文。

因此，第 9 节中的人物规则已有答案：人物与服装逐条变化，账号整体审美保持年轻、自然、
干净、生活化，场景与人物状态随文案变化。

### 11.2 “只输入品类”仍存在的事实缺口

文案模板只允许选择 1–2 个与当前场景相关的使用动作或可观察细节，但当前最简输入只有
【商品】或品类。如果输入仅为“雨靴”，模型仍无法知道具体商品的材质、做工、款式、
包装或功能；让模型自行补齐会与“不夸张、不虚假”冲突。

建议把输入分成两种模式：

| 模式 | 输入 | 输出边界 |
|---|---|---|
| 品类草案模式 | 仅输入“雨靴”等品类 | 只使用通用场景和低风险描述，必须人工复核 |
| 商品生产模式 | 商品名 + 经确认的商品介绍/卖点 | 可生成准备投放的文案与数字人任务 |

推荐生产输入最少包含：

```yaml
商品: 浅卡其色中筒雨靴
品类: 雨靴
商品介绍:
  - 浅卡其配色
  - 中筒款式
  - 日常雨天穿着
不可使用的信息:
  - 未提供的具体材质
  - 防滑等级
  - 绝对防水承诺
```

注入文案 Prompt 时，【商品】应包含“商品名 + 已确认商品介绍”，而不是只传一个宽泛
品类。这样才能同时满足真实卖点和禁止虚构的要求。

### 11.3 与现有 `Auto Oceanengine` 的关键接口差异

经核对现有实现，目标项目当前的真实链路是：

```text
person_prompt
  → 即创图片模型生成一张人物图
  → 登记为人物素材

script
  → TTS + 已登记人物图
  → 配音生视频
```

也就是说，CSV 中的 `person_prompt` 当前进入的是**人物图片生成接口**；视频创建接口只
接收人物图片、音色和 `script`，没有接收第二套“完整数字人视频 Prompt”的字段。

这带来两个结论：

1. 已验证的数字人 Prompt 可以作为上游视觉导演 Prompt，但其“口型同步、眨眼、一镜
   到底、不运镜”等视频动作指令，在当前链路中不会直接传给视频模型；
2. 若把完整数字人视频 Prompt 原样写入 `person_prompt`，图片模型通常只能落实人物、
   场景、服装、商品摆放、构图、光线和摄影风格，动作与时序指令可能被忽略。

首版建议从数字人 Prompt 中派生两个结果：

```text
完整数字人 Prompt
  ├─ 归档：保留完整生成结果，供未来即梦/可灵/Veo/Seedance/Runway 使用
  └─ 图片版 person_prompt：提取首帧可见信息，写入现有即创 CSV
```

“图片版 `person_prompt`”应只保留静态可见内容：

- 当前条目的独立人物外观；
- 服装、发型、表情和姿态；
- 单一生活场景；
- 商品位置与展示方式；
- 9:16、中景或半身、正面构图；
- 自然光、手机实拍、真实肤质等画面风格；
- 无字幕、Logo、水印、直播间和广告棚。

### 11.4 多人物与服装差异

现行规则不再要求固定同一张脸。每条 Prompt 必须改变脸型、可见五官特征、发型、发色和
完整服装组合；批次内使用 `identity_key` 与 `outfit_key` 做唯一性检查。不同人物仍保持
22–26 岁亚洲女生、邻家女孩气质、自然干净和生活化的统一账号审美。

禁止在人物 Prompt 中出现“与上一条相同”“固定人物”“保持脸部特征一致”等连续性表述。

### 11.5 修订后的推荐数据流

```text
用户输入商品/品类资料
  → 注入已验证文案 Prompt
  → 80–100 字成品口播
  ├─ Prompt Engineering/<YYYYMMDD>/<task>/<task_id>.smartsplit.txt
  └─ 注入已验证数字人 Prompt
       → 完整数字人视频 Prompt（留档）
       → 静态画面信息提取
       → Prompt Engineering/<YYYYMMDD>/<task>/<task>.csv
       → 用户另行要求后执行 preflight
       → 人工确认
       → import / run-api-video
```

建议上游记录增加两个独立字段，避免混淆：

| 字段 | 用途 |
|---|---|
| `avatar_video_prompt` | 已验证数字人模板生成的完整视频 Prompt，供归档及其他视频模型使用 |
| `person_prompt` | 从完整 Prompt 提取的静态人物图 Prompt，供现有即创项目使用 |

目标项目的 CSV 暂不需要增加字段；`avatar_video_prompt` 保留在上游生成记录中即可。

## 12. 当前最值得先确认的问题

在不改代码的阶段，剩下的核心选择已经收敛为两个：

1. **商品资料来源**：实际使用时是否除了品类，还会提供商品标题、详情页文案、商品图或
   人工整理的真实卖点？如果始终只输入品类，首版只能定位为创意草案工具。
人物策略已确定为每条不同人物和不同服装；剩余核心问题是商品资料来源。这个答案会直接
决定首版的数据结构和事实边界，其他部分已经具备实施条件。
